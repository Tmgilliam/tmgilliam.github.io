# Public Proof Gate

A public release is permitted only when every applicable item is satisfied.

## Value

- [ ] The target user is named.
- [ ] The problem is understandable in 30 seconds.
- [ ] The repository provides one hero use case.
- [ ] The output is useful without access to private BBIOS data.

## Usability

- [ ] Quick start takes five minutes or less.
- [ ] A working example is included.
- [ ] Expected output is shown.
- [ ] Failure behavior is documented.

## Evidence

- [ ] Tests pass.
- [ ] CI passes.
- [ ] Claims trace to code, tests, examples, sources, or labeled assumptions.
- [ ] Limitations and non-goals are explicit.
- [ ] Static examples are not represented as live telemetry.

## Safety and governance

- [ ] No secrets, personal data, client data, employer-confidential material, or private Nova content.
- [ ] Human-approval boundaries are explicit where relevant.
- [ ] Security reporting is documented.
- [ ] Legal, compliance, and certification claims have been reviewed.
- [ ] License has been selected and added.

## Maintenance

- [ ] Version number exists.
- [ ] Changelog exists.
- [ ] Release notes are drafted.
- [ ] Contribution rules exist.
- [ ] Support expectations exist.
- [ ] At least one beginner-friendly issue is available.

## Distribution

- [ ] Repository description and topics are set.
- [ ] Social preview image is added.
- [ ] One launch post, article, or video is ready.
- [ ] The repository is pinned if it is among the strongest public proofs.

## Portfolio site application

This Docusaurus site follows the same boundary rules:

- Documentation describes **public repositories** and **labeled portfolio patterns**
- Blog posts distinguish **published** articles from **drafts**
- ERP and warehouse content uses sanitized methodology—not client or employer identifiers
- Certification claims match completed credentials (see [Certifications](/docs/certifications/))

See also [ADR-002: Public Proof Boundary](/docs/architecture-decisions/adr-002-public-proof-boundary).
